<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client→Supabase | Untrusted user input (email, password, form data) crosses here |
| Supabase→client | Supabase responses (session, data) received here |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-01 | Injection | AuthModal (form input) | mitigate | Validate email format with regex, password min 6 chars; Supabase handles SQL injection |
| T-01-02 | Spoofing | Auth flow | mitigate | Use Supabase session tokens; RLS will protect user data |
| T-01-03 | Tampering | Dashboard data | mitigate | RLS policies ensure users only see their own data |
| T-01-04 | Info Disclosure | Auth responses | mitigate | Don't expose sensitive error details to client; generic "Invalid credentials" |
| T-01-05 | Denial of Service | Auth endpoint | accept | Supabase handles rate limiting; low risk for MVP |
</threat_model>
