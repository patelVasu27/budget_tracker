<action>
Create `src/components/ErrorBoundary.tsx` as a React class component (error boundaries must be class components тАФ function components cannot implement `componentDidCatch`).

Structure:
- `Props`: `{ children: ReactNode }`
- `State`: `{ hasError: boolean; error: Error | null }`
- `static getDerivedStateFromError(error: Error): State` тАФ captures error into state
- `componentDidCatch(error: Error, errorInfo: ErrorInfo)` тАФ logs to `console.error` only (no external reporting for MVP)
- `handleReset` тАФ sets `{ hasError: false, error: null }` so the app re-renders children

In `render()`:
- If `hasError` is true: render full-screen fallback UI:
  - `role="alert"` on outer container (accessibility)
  - `AlertTriangle` icon from lucide-react in a red-tinted circle
  - Title: "Something went wrong" (`<h1>`)
  - Description: "SimpleBudget encountered an unexpected error. Please try again." (`<p>`)
  - "Try Again" button calling `handleReset` тАФ styled with `btn-primary` class
  - Centered, padded layout matching the existing app background (`bg-background`)
- If no error: render `this.props.children`
- Do NOT expose `error.message` or stack trace in production UI (Information Disclosure тАФ per threat model)
- Do NOT import `lucide-react` client-side тАФ destructure `AlertTriangle` from `lucide-react` import

In `src/main.tsx`:
- Add `import { ErrorBoundary } from './components/ErrorBoundary'`
- Nest `<App />` inside `<ErrorBoundary>` wrapper, keeping `<QueryClientProvider>` as the outer provider
- Result structure: `<React.StrictMode><QueryClientProvider><ErrorBoundary><App /></ErrorBoundary></QueryClientProvider></React.StrictMode>`

<action>
Create `.github/workflows/ci.yml` at the project root (create `.github/workflows/` directories if they don't exist).

Trigger: `push` and `pull_request` on `main` branch.

The workflow must contain a single `ci` job:

```yaml
name: CI/CD

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  ci:
    runs-on: ubuntu-latest

    steps:
      - name: Checkout
        uses: actions/checkout@v4

      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: 'npm'

      - name: Install dependencies
        run: npm ci

      - name: TypeScript check
        run: npx tsc --noEmit

      - name: Run tests
        run: npx vitest run

      - name: Deploy to Vercel
        if: github.ref == 'refs/heads/main' && github.event_name == 'push'
        uses: amondnet/vercel-action@v25
        with:
          vercel-token: ${{ secrets.VERCEL_TOKEN }}
          vercel-org-id: ${{ secrets.VERCEL_ORG_ID }}
          vercel-project-id: ${{ secrets.VERCEL_PROJECT_ID }}
          vercel-args: '--prod'
```

Key design choices:
- `npm ci` instead of `npm install` тАФ fails if `package-lock.json` is out of sync, enforces reproducible builds
- `npx tsc --noEmit` before `npx vitest run` тАФ fastest failure mode; if types fail, tests won't compile anyway
- `amondnet/vercel-action@v25` тАФ battle-tested GitHub Action for Vercel deployment; uses `--prod` for production
- Guard on `github.ref == 'refs/heads/main' && github.event_name == 'push'` тАФ only deploys on push to main, not on PR
- Vercel secrets (`VERCEL_TOKEN`, `VERCEL_ORG_ID`, `VERCEL_PROJECT_ID`) documented in `user_setup` frontmatter тАФ user must configure these in GitHub repo Settings тЖТ Secrets and variables тЖТ Actions before first deployment

Edge cases considered:
- PR from fork: `npm ci` + `tsc` + `vitest` still run (the deploy step is skipped because `github.ref` won't be `main`)
- Manual deploy: user can still use Vercel CLI directly тАФ this CI is additive, not a replacement
</action>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| ErrorBoundary тЖТ console | Console output in production (debug info leak) |
| CI/CD secrets тЖТ Actions runner | VERCEL_TOKEN, ORG_ID, PROJECT_ID in GitHub Secrets |
| ErrorBoundary тЖТ user | Error UI visible in browser |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-01 | Information Disclosure | ErrorBoundary | mitigate | Render only generic fallback text in production тАФ never expose `error.message`, `error.stack`, or `componentStack` to the user. Log details to console only. |
| T-04-02 | Elevation of Privilege | CI/CD workflow | accept | VERCEL_TOKEN has deployment-only scope. If leaked, worst case is unauthorized deploys тАФ no data access. Token is stored as GitHub encrypted secret, never logged. |
| T-04-03 | Tampering | CI/CD workflow | mitigate | `npm ci` enforces lockfile integrity (fails if `package-lock.json` doesn't match `package.json`). No action pins to mutable tags тАФ `actions/checkout@v4`, `actions/setup-node@v4`, `amondnet/vercel-action@v25` use major version tags. |
| T-04-04 | Spoofing | Vercel deploy | transfer | Vercel's `--prod` flag + token auth ensures only authenticated deploys from the CI pipeline succeed. Access to the Vercel token is limited to repo admin. |
</threat_model>
